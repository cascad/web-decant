# decant — архитектура и варианты захвата

Документ описывает, как устроен decant, и **два варианта «фронта захвата»**:
- **Вариант A** — текущий, «чистый» (букмарклет + Playwright-демон);
- **Вариант B** — через браузерное расширение (+ Native Messaging).

Плюс план по **вложениям (PDF / Office)** — раздел в конце.

---

## Принцип

decant не борется с авторизацией Confluence/SSO/2FA, а **переиспользует уже залогиненную браузерную сессию**. Понимание (что изображено на схеме, текст внутри PDF-скана и т.п.) **сознательно вынесено** из захвата в отдельный шаг: decant — это **экстрактор**, а не «понималка».

Ключевая мысль про агентов: MCP/CLI тут — **не к Confluence**, а **к локальному снимку страницы**. Это и обходит корпоративные ограничения, и не требует выдавать токены/полный доступ.

---

## Общий каркас (одинаков для обоих вариантов)

Что бы ни стояло «фронтом», ниже всё одинаково:

- **Локальный демон** (`serve.py`, `127.0.0.1:8765`, токен-гейт) — принимает снимок страницы, пишет бандл на диск.
- **Бандл на странице:**
  ```
  captures/<slug>/
    <slug>.md        # чистый markdown + локальные ссылки на медиа в местах вставки
    media/           # извлечённые байты (картинки/схемы/видео; позже — вложения)
    media.json       # манифест: что, где стояло, + пустые поля под обработку
  ```
- **Экстракция текста:** trafilatura → markdown (с `markdownify`-fallback).
- **Извлечение медиа** (`media.py`): только **байты + позиции**; понимание отложено (`description/ocr/mermaid = null`).
- **Агент-интерфейс:** CLI (`decant get` / `decant current`); MCP к локальному снимку — в роадмапе.

`media.json` — это **контракт** между decant (извлечение) и отдельным агентом/сетью (понимание). decant заполняет факты об извлечении, поля `processed/description/ocr/mermaid` оставляет пустыми.

## Вариант A — текущий («чистый»): букмарклет + Playwright-демон

**Как работает.** Браузерная сессия живёт в **выделенном персистентном профиле** Playwright (логинишься один раз, 2FA включительно). Захват инициируется одним из двух способов:
1. **`decant get <url>`** — демон headless рендерит страницу в своей сессии, извлекает текст и медиа.
2. **Букмарклет** «Rip this page» — на странице шлёт демону **живой DOM** (`POST text/plain`, без CORS-preflight). На сайтах с жёстким CSP (как Confluence) `fetch` на localhost блокируется → откат: букмарклет открывает вкладку демона, и тот **до-качивает страницу своей сессией** (Playwright-путь).

**Медиа** извлекаются только на **Playwright-пути** (нужна живая страница для авторизованных загрузок, `element.screenshot`, сериализации SVG и определения позиций). Live-DOM-путь букмарклета медиа не делает.

**Плюсы**
- **Ноль установки и подписи** — букмарклет это просто закладка; работает в **Firefox** (где расширения требуют подписи Mozilla).
- `element.screenshot` в Playwright — **чистый** per-element скриншот схемы (без возни с координатами/devicePixelRatio).
- Изолированный профиль: воспроизводимо, не мешает обычному браузеру, легко раздать.

**Минусы**
- На CSP-залоченном Confluence букмарклет деградирует в демон-рефетч → это **отдельная сессия** (нужен разовый `decant login`), а не ровно текущая вкладка.
- Букмарклет нельзя повесить на глобальный хоткей (жмётся мышкой).

## Вариант B — браузерное расширение + Native Messaging

**Как работало бы.** Расширение Chrome/Edge с кнопкой/хоткеем «Capture»:
- `activeTab` + content script читают **текущую реальную вкладку** пользователя: DOM, видимый текст, ссылки, `img/src`, скриншоты.
- Снимок уходит локальному демону через **Chrome Native Messaging** (extension ↔ native host), **минуя CSP страницы** — расширение не подчиняется `connect-src`.
- Дальше — тот же демон и тот же бандл, что в варианте A.

**Плюсы**
- **CSP-устойчиво:** Native Messaging не блокируется политикой страницы (в отличие от `fetch` букмарклета). На закрытом Confluence работает «в лоб».
- **Снимает именно текущую вкладку** без отдельного логина — ровно то, что видит пользователь, включая интерактивное/сессионное состояние.
- Можно повесить на **горячую клавишу** и контекст-меню.

**Минусы**
- **Chrome/Edge-only**; у нас пользователь на **Firefox** (можно и FF-расширение, но это другой манифест + подпись).
- **Сборка + подпись + установка** расширения (для Firefox — подпись Mozilla; для Chrome — упаковка/стор или dev-режим).
- **Per-element скриншот в расширении неудобен:** `captureVisibleTab` даёт вьюпорт, элемент надо вырезать по `getBoundingClientRect` с учётом скролла и devicePixelRatio — в Playwright это одна строка.

## Сравнение A vs B

| | **A — букмарклет + Playwright** (текущий) | **B — расширение + Native Messaging** |
|---|---|---|
| Установка | ноль (закладка) | сборка + подпись + установка |
| Браузер | любой, в т.ч. **Firefox** | Chrome/Edge (или отдельный FF-аддон) |
| CSP-Confluence | `fetch` блокируется → демон-рефетч (отд. сессия) | **минует CSP** (Native Messaging) |
| Что снимает | живой DOM или рендер демона | **точно текущую вкладку** пользователя |
| Доп. логин | разовый `decant login` для рефетча | не нужен (та же сессия) |
| Скриншот схемы | **чисто** (Playwright `element.screenshot`) | вьюпорт + кроп по координатам |
| Хоткей | нет (клик по закладке) | да |

**Вывод.** B надёжнее на залоченном Confluence и приятнее по UX, но дороже (Chrome-only, сборка/подпись). A выбран как стартовый из-за Firefox и нулевой установки. Демон/бандл/манифест у обоих **общие** — переключение фронта не ломает остальное.

## Медиа (как сейчас работает)

`media.py`, по умолчанию включено (`--no-media` отключает). Только Playwright-путь.

- **Картинки:** скачиваем оригинал по `src` авторизованным запросом (полное разрешение); fallback — `element.screenshot`.
- **SVG-схемы** (draw.io/Gliffy часто SVG): сериализуем XML (`original_file`) + png-рендер (`file`).
- **canvas / диаграммы-макросы:** `element.screenshot`.
- **Видео:** скачиваем файл из `<video><source>`; внешние эмбеды (YouTube/Vimeo) — заглушка в манифесте (только `source_url`).
- Дедуп по `sha256`; фильтр иконок по размеру.

**Позиции — двухуровнево:** что вшилось в текст — inline `![](media/..)` + якорь `<!-- decant:media id=.. -->`; что экстрактор текста уронил — в аппендикс `## Медиа (вне текста)` с `after_heading`/`preceding_text` из манифеста.

## Вложения: PDF / Office (план, ещё НЕ реализовано)

**Пробел.** Confluence-страницы часто несут **прикреплённые документы** — PDF, Word (`.docx`), Excel (`.xlsx`), PowerPoint (`.pptx`), а также макросы «View file»/«Attachments». Сейчас decant их **не конвертирует** — они остаются ссылками.

**Предлагаемый слой (тот же принцип «извлечь, не понимать»):**
1. **Детект** вложений: ссылки `/download/attachments/...`, макросы view-file, `data-linked-resource-type`.
2. **Скачать байты** авторизованной сессией (как медиа) → в `media/` (или `attachments/`).
3. **Структурно сконвертировать в markdown** через **[MarkItDown](https://github.com/microsoft/markitdown)** (Microsoft): PDF / docx / xlsx / pptx / html / изображения → markdown. Это **детерминированная** конвертация, поэтому честно относится к слою экстракции (не «понимание»).
4. **Сохранить** рядом: оригинал + `*.md`-сайдкар + запись в манифесте.
5. **Позиция** — как у медиа: inline-ссылка на месте + `occurrence` в манифесте.

**Манифест** расширяется записью вида:
```jsonc
{
  "id": "att-001",
  "kind": "attachment",
  "format": "pdf",                 // pdf | docx | xlsx | pptx | ...
  "file": "media/att-001.pdf",     // оригинал
  "text_file": "media/att-001.md", // результат MarkItDown (детерминированный текст)
  "source_url": "https://confluence.../download/attachments/123/spec.pdf",
  "occurrences": [ { "order": 12, "after_heading": "## Спецификация" } ],
  "processed": false,              // ↓ под отдельный агент/сеть
  "description": null, "ocr": null
}
```

**Нюансы:**
- **Текстовый PDF/Office** → MarkItDown даёт текст детерминированно (этап экстракции).
- **Скан/картиночный PDF** → текста нет; OCR/описание оставляем на **отдельный шаг** (`ocr/description` в манифесте), как и для картинок.
- Слой **ортогонален фронту захвата**: и вариант A, и вариант B просто отдают демону URL вложения, конвертация — на сервере (Python), `pip install markitdown`. При желании — тонкий контроль через `pdfplumber`/`python-docx`/`openpyxl`.

## Статус и роадмап

**Готово:** `get` / `serve` / `bookmarklet` / `current`; извлечение медиа + `media.json`; кроссплатформенные инсталлеры. Вариант захвата — **A**.

**Роадмап:** вложения PDF/Office (MarkItDown-слой); соседние страницы (`crawl --depth`); MCP-сервер к локальному снимку; опционально — вариант **B** (расширение) как более устойчивый фронт; отдельный обработчик, заполняющий `description/ocr/mermaid` в манифестах.
